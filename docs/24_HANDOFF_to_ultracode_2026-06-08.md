## 1. 확정 결정 (위배 금지)

| # | 결정 | 값 |
|---|---|---|
| 1 | 전략 앵커 | "경쟁사 빈칸" ❌ → **글쓰기 학습 중심화라는 진짜 수요.** 가장 잘 보조 + 교사 인정 + 학생 자유 사용. 아웃컴·데이터는 결과지 깃발 아님 |

| 4 | 핵심 루프 | 5단계 전부: ①과제·루브릭 해독 ②소크라테스식 끌어내기 ③초안 함께쓰기(발판) ④고쳐쓰기 루프 ⑤성찰+과정증거 |

| 6 | 1차 유통 | **교사 인정 주도** (교사 무료 도구=루브릭 등록+과정로그 열람; 채점 자동화는 안 만듦) → 학생 자유 사용 → 가계 전환 |

**제품 원칙(전 단계 공통):** *생각·구조·고쳐쓰기는 끌어내되, 문장은 대신 쓰지 않는다.* 이게 실력↑·교사인정·자유사용을 동시 성립시키는 축. **무너지면 ChatGPT와 같아짐.**

**경쟁:** 디폴트=KEEwi(투블럭AI, 평가기). 우리는 과정 코치 → *결과 vs 성장* 다른 축. 정면 채점경쟁 회피.

## 2. 확정된 UX 결정 (지금까지 깎은 것)

- **레이아웃 = 바텀시트 코치 (모바일·태블릿 우선).** 글이 화면 차지(캔버스), 코치는 하단 시트에서 **한 호흡에 한 nudge**만 peek → 쓸어올리면 펼침. 챗봇 입력창 ❌. (인라인 밑줄 신호는 *나중에 얹는 보조 레이어*로 보류.)
  ```
  ┌──────────────────┐
  │ 화산의 형성    ⋯ │
  │ 화산은 마그마가   │  ← 캔버스(글, 주인공)
  │ 분출해 만들어진   │
  │ 지형이다...      │
  │╭────────────────╮│
  ││💬 이 문단, 근거가││  ← 코치 시트(peek)
  ││ 비었어.  쓸어올려↑││    1 nudge only
  │╰────────────────╯│
  └──────────────────┘
  ```
- **고쳐쓰기 루프 인터랙션 (제품 심장):**
  1. **nudge peek** — 가장 점수 끌어올릴 1개만 ("근거가 비었어. 쓸어올려↑")
  2. **펼침** — 진단 + *유도 질문* + 루브릭 영역. 대신 안 써줌:
     `💬 내용 충실도·2번째 문단 / 주장만 있고 근거 없어 / ❓네 경험에서 '왜'를 하나 / [이 문단 고치기][왜 중요해?]`
  3. **학생이 캔버스에서 직접 고침** (시트 접힘)
  4. **재점검 → 성장 표시 → 다음 nudge**
- **④ 성장 피드백 = "성장 막대(전/후)".** 수치 점수 노출 대신 영역별 막대가 차오르는 시각으로 "차올랐다"를 직관 표현(평가기 점수집착 회귀 회피).
  ```
  ╭────────────────╮
  │ 내용 충실도      │
  │ 전 ▰▰▱▱▱        │
  │ 후 ▰▰▰▰▱ ↑      │
  │ [다음 ▸]        │
  ╰────────────────╯
  ```
- **재점검 트리거 = 명시 버튼.** 학생이 고친 뒤 **"다 고쳤어 ✓"** 누를 때만 코치가 재평가(자동 재평가 ❌). 학생 통제·감시 느낌 제거·모델 호출 비용 예측 가능. (자동/하이브리드는 v2 후보)
- **nudge 우선순위 = 큰 향상·쉬운 것 먼저(quick win).** 여러 영역이 약할 때 "점수 끌어올림 폭이 크고 고치기 쉬운" 것부터 띄움 → 초반 성공체험으로 탄성 → 성장 막대가 크게 움직임. (배점순/흐름순/학생선택 ❌)
- 톤: 주니어 푸리와 달리 **담백한 코치 톤**(중고생). 과부하 방지: 한 번에 한 가지. OS 셸 토큰(blue+lemon) 락. ※radius 토큰은 이 레포에 없음(`--r-*`는 pullim-web) — 출처 확인 후 정합.

## 3. 아직 안 정한 것 (계획에 "결정 필요"로 표시할 것 — 임의 확정 금지)

**UX:**
- 성장 막대의 기준 점수 산출 방식(루브릭 0~20 매핑을 막대로 — 내부 수치는 쓰되 노출만 막대)
- quick-win 우선순위의 "고치기 쉬움" 추정 방식(휴리스틱 vs 모델 판단)
- 인라인 밑줄 보조 레이어 도입 시점
- 데스크톱/태블릿 split 뷰 변형

**전략(기본값으로 두되 확정 필요):**
- 가격(기본값 월 2.9~4.9만, WTP 검증 후 확정)
- 중1(만13) 처리(기본값: 보호자 동의 별도 트랙, 만14+ 우선 출시)
- 단품 SAM bottom-up 수치
- 교사 무료/유료 경계

## 4. 현재 코드 상태 (greenfield 아님 — 위에 얹어라)

- **이미 동작하는 것:** 라이브 `POST /api/score`(Haiku 4.5, 토큰게이트, 정규화, 후처리 가드, 스키마 재호출, Sentry, rate-limit). FE 위저드 `/try`, `/samples`, `/results`(이력), `/me`, `/onboarding`, export, 다크모드. E2E(playwright)+컴포넌트(vitest) CI. Vercel `main` 자동배포(pullim-demo.vercel.app).
- **재사용 핵심:** `app/lib/grading.ts`(순수 채점·가드 — SDK import 금지 규칙), `app/lib/prompt.ts`(프롬프트 단일소스), `app/data/samples.ts`(F3Output 타입·5종 anchor), `app/components/ScoreForm.tsx`·`ResultView.tsx`.
- **"대신 안 써준다" 가드**는 `grading.ts` 기존 후처리 가드 패턴을 확장해 구현 (생성/문장완성 출력 차단 + 질문·진단·발판만 허용).
- **전환의 본질:** 현재는 "다 쓴 글 1회 채점"(평가기) → "쓰는 동안 + 고쳐쓰기 반복 코칭"(과정 코치). 즉 **세션·드래프트 상태·문단 단위 진단·바텀시트 UI·교사 루브릭 입력·과정 로그**가 신규 영역.

## 6. ultracode에게 요청하는 산출물

> 전략 재설계 금지. 아래만.

1. **"평가기 → 과정 코치" 구현 계획** — 에픽/티켓 분해, 의존성, 위험.
2. **90일 MVP 우선순위:** 킥(②소크라테스 끌어내기 + ④고쳐쓰기 루프) + ①루브릭 해독 + 바텀시트 코치 UI + "생성 차단" 가드 + 교사 무료 도구(루브릭·과정로그) 최소판.
3. **아키텍처 결정 제안:** 세션/드래프트 상태 모델, 문단 단위 진단 파이프라인(기존 `/api/score` 재사용 vs 신규 코칭 엔드포인트), 바텀시트 컴포넌트 구조, 주니어 엔진 공유를 위한 순수 모듈 경계.
4. **§3 미정 항목은 "결정 필요"로 표시** — 임의 확정하지 말 것.
5. 기존 자산 재사용 vs 신규 구축을 명시.

## 0. 한 줄 결론

**구조·재현성·강제 룰은 통과, 그러나 EPO 정답 점수를 재현하지 못했고(상위권 −12~−24점) 응답 시간이 목표(15초)를 크게 초과(~25초)했다. → 프롬프트 v0.2 보정 필요. freeze는 점수 보정 전까지 보류.**

## 1. 종합 판정

| 기준 | 결과 | 비고 |
|---|---|---|
| 스키마 유효 (25/25) | ✅ PASS | 5영역·총점=합·guides 1~3 모두 충족 (코드펜스 제거 후) |
| **재현 분산** (같은 글 재호출 점수 안정성) | ✅ PASS | 총점 run-to-run 폭 ≤2, 영역 폭 대부분 ≤3 — 매우 안정적 |
| **FIX_COUNT_CHECK** (C-1 회귀 방지) | ✅ PASS | 25건 중 위반 0건 — 신설 강제 룰 작동 |
| **정답 ±3점 (총점)** | ❌ FAIL | 상위권에서 체계적 하향 편향 (B −24, E −13, A −12) |
| **정답 ±2점 (영역)** | ❌ FAIL | 전 영역 하향, 특히 내용·구조·성장 |
| **응답 시간 15초 목표** | ❌ FAIL | median ~25~29초 (functional_spec §3 F3 / §8 Q4) |
| 출력 형식 ('JSON only, 펜스 금지') | ⚠️ 부분 위반 | 모델이 ```json 코드펜스를 자주 붙임 — 파서가 흡수해 스키마는 통과 |
| DUPLICATION_CHECK | ⚠️ heuristic 경고 | C·B에서 정정 토큰이 fix·guide 양쪽 등장 — 룰 미준수 의심 |

> **HARD PASS = ❌ FAIL.** 단, 실패의 성격이 중요하다 — 아래 §2.

## 2. 핵심 발견 — "일관되지만 박하다"

### 2.1 재현성은 우수 (의도대로 작동)

temperature 0.2에서 같은 글을 5회 채점한 총점 분산:

| 샘플 | 모델 총점 5회 | 폭 | 정답 |
|---|---|---:|---:|
| D | 44·42·44·44·44 | 2 | 40 |
| C | 62·62·63·63·62 | 1 | 61 |
| A | 61·62·62·62·62 | 1 | 74 |
| E | 72·72·72·72·72 | 0 | 85 |
| B | 62·62·62·62·62 | 0 | 86 |

→ functional_spec §7.1 위험("같은 글 다른 점수")은 **해소**. 프롬프트는 결정적으로 재현된다.

### 2.2 그러나 정답(EPO) 점수를 재현 못 함 — 상위권일수록 격차 확대

| 샘플 | 정답 | 모델 median | 격차 |
|---|---:|---:|---:|
| D (저점) | 40 | 44 | **+4** |
| C (편차) | 61 | 62 | +1 ✅ |
| A (중점) | 74 | 62 | **−12** |
| E (중상) | 85 | 72 | **−13** |
| B (고점) | 86 | 62 | **−24** |

**패턴**: 글 품질이 높을수록 모델이 더 박하게 준다. 고품질 글(A 74·B 86)이 모두 ~62로 수렴 — 모델이 상위 구간을 압축한다.

### 2.3 영역별 진단 (B·A·E 1회 호출 상세)

| 영역 | B 정답→모델 | A 정답→모델 | E 정답→모델 |
|---|---|---|---|
| 과제 이해 | 18→14 | 15→14 | 18→14 |
| 내용 충실도 | 18→**12** | 15→14 | 17→14 |
| 구조·논리 | 16→**10** | 14→12 | 17→14 |
| 표현·문장 | 16→**10** | 15→12 | 16→15 |
| 성장 가능성 | 18→16 | 15→**10** | 17→15 |

**해석**:
- **내용 충실도·구조·논리**에서 가장 크게 깎인다(B 각 −6). 모델은 EPO보다 '다층위 근거'·'결론 완결성' 기준을 엄격히 적용한다. 특히 B의 mechanics 오타('구가간' 등)+장르 혼합(동기 진술 결론)을 EPO(18/16)보다 훨씬 무겁게(12/10) 본다.
- **성장 가능성**도 박하다(A 15→10). 모델은 '한 번의 수정으로 닿는가'를 보수적으로 판단.
- 즉 **rubric 앵커를 그대로 인코딩했더니 모델이 EPO보다 일관되게 엄격**해졌다. EPO 정답이 후했거나, 앵커 문구가 상위 구간 도달을 어렵게 만든다.

### 2.4 응답 시간 — 15초 목표 초과

| 샘플 | median ms |
|---|---:|
| D | 24,906 |
| C | 28,938 |
| A | 25,671 |
| E | 24,968 |
| B | 25,830 |

→ ~25초. functional_spec §3 F3 "15초 이내" 목표 **미달**. (키 점검용 ping은 4토큰이라 1.4초였으나, 실제 채점은 600~900토큰 생성이라 ~25초.) §8 Q4("15초 현실적인가")의 답: **현 구성(Sonnet 4.6, max_tokens 2000)에선 비현실적.**

## 3. 부수 발견

1. **코드펜스 위반**: 모델이 출력을 ```json … ``` 으로 감싸는 경우가 잦다(진단 3건 모두). 프롬프트의 "마크다운 코드펜스 금지"를 어김. 클라이언트 파서가 펜스를 제거해 스키마 검증은 통과하므로 치명적이지 않으나, v0.2에서 (a) 지시 강화 또는 (b) 클라이언트 펜스 제거를 정식 처리로 명문화.
2. **DUPLICATION 미준수 의심**: C('가김기'·'초짐')·B('구가간'·'이글')에서 정정 토큰이 feedback_fix와 revision_guides 양쪽에 등장(heuristic). FIX_COUNT는 지켰으나(fix에 4건 미만) DUPLICATION_CHECK는 덜 지킨다. 프롬프트만으로는 한계 → 클라이언트 후처리 가드 필요(08 §7 #2).

## 4. 결론과 v0.2 처방

### 4.1 결론
- 프롬프트 v0.1은 **재현성·스키마·FIX_COUNT(C-1 회귀 방지)** 면에서 작동한다.
- 그러나 **EPO 점수 보정 실패(상위권 −12~−24)**와 **응답 시간 초과(~25초)**로 freeze 기준 미달.
- 이 검증의 가치: freeze 전에 **'일관되지만 EPO와 캘리브레이션이 안 맞는다'**는 사실을 데이터로 드러냄. (10 freeze 요청 §1.5 #1에 대한 답)

### 4.2 v0.2 처방 (우선순위)

| # | 처방 | 근거 |
|---|---|---|
| 1 | **few-shot 캘리브레이션** — 08 §4의 5건(정답)을 시스템 프롬프트 끝에 few-shot anchor로 추가. 모델 점수를 EPO 분포로 끌어올림 | §2.2 체계적 하향 편향. 08 §4가 이미 fallback으로 예고 |
| 2 | **상위 구간 앵커 재서술** — 18~20/15~17 도달 조건을 모델이 너무 엄격히 읽지 않게 완화 표현. 또는 EPO 정답 자체를 박승훈·김아진과 재합의(어느 쪽이 '맞는' 점수인지 결정) | §2.3 내용·구조에서 −6 |
| 3 | **응답 시간** — Haiku 4.5로 재측정 / max_tokens 축소(출력 실측 600~900) / 스트리밍 검토. 안 되면 §3 F3 목표를 20~30초로 현실화 | §2.4 |
| 4 | **코드펜스·DUPLICATION** — 클라이언트 후처리 정식화(펜스 제거 + 'A→B' 카운트/중복 가드) | §3 |

### 4.3 freeze 영향
- 10 freeze 요청 §1.5 #1 → **결론: 재현성은 ✓이나 점수 보정 ✗.** 두 분(박승훈·김아진)께 "프롬프트 v0.2 보정 후 재검증"을 freeze 전제로 권고. 또는 **rubric/샘플은 freeze하되 프롬프트는 v0.2까지 별도 트랙**으로 분리하는 안.
- ⚠️ 중요한 정책 결정: **A 74·B 86 같은 EPO 정답이 옳은가, 모델의 62가 옳은가**를 박승훈·김아진이 판정해야 한다. 이건 프롬프트 버그가 아니라 **채점 철학(상위 구간을 얼마나 후하게 줄까)**의 문제다.

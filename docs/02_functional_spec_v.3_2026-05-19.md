## 1. 범위 — 확정된 7개 기능

본 명세서의 F1~F7은 화면 기능 단위다. 기능제안서(`feature_proposal_mvp_candidates_2026-05-14.md`) §5.1 기능 구조의 F-코드와 아래와 같이 매핑된다.

| ID | 기능 | 한 줄 정의 | 기능제안서 F-코드 |
|---|---|---|---|
| F1 | 과제 설명 입력 | 학생이 과제 조건(과목·학년·장르·분량·과제문)을 입력한다 | F01 과제 조건 입력 폼 · F09 학교급·과목 프리셋 |
| F2 | 학생 글 입력 | 학생이 본문 글을 붙여넣고 글자 수를 확인한다 | F02 |
| F3 | AI 첨삭 실행 | 입력값을 AI에 전달해 채점·피드백·수정가이드를 생성한다 | (실행 레이어 — F03·F04·F05·F12 생성 오케스트레이션) |
| F4 | 5영역 점수 표시 | 5영역 각 0~20점, 총 100점을 화면에 표시한다 | F03 · F12 |
| F5 | 영역별 피드백 표시 | 영역별로 잘한 점·고칠 점을 문장 단위로 표시한다 | F04 (F10 과목·영역 태깅은 Month 2) |
| F6 | 수정 가이드 표시 | 다음 수정에서 바로 반영할 1~3개 액션을 표시한다 | F05 |
| F7 | 결과 복사 기능 | 점수+피드백+수정가이드를 텍스트로 클립보드에 복사한다 | F06 · F08 (F13 수정 전/후 비교는 Month 2) |

**v0.1 범위 외**: F10 교육과정 과목·영역 태깅, F13 수정 전/후 비교는 기능제안서에서 P1(Month 2)로 확정 — 본 명세서는 다루지 않는다.

**화면 구성**: 단일 페이지(SPA) 1개. 입력 영역(F1·F2) → 실행 버튼(F3) → 결과 영역(F4·F5·F6·F7).

## 2. 사용자 흐름

```text
[1] 페이지 진입
     ↓
[2] 과제 설명 입력 (F1) — 과목/학년/장르/분량/과제문
     ↓
[3] 학생 글 붙여넣기 (F2) — 본문 + 글자 수 자동 표시
     ↓
[4] "AI 첨삭 받기" 버튼 클릭 (F3)
     ↓
[5] 로딩 표시 (3~15초)
     ↓
[6] 결과 표시
     ├─ 총점 + 5영역 점수 (F4)
     ├─ 영역별 피드백 (F5)
     └─ 수정 가이드 1~3개 (F6)
     ↓
[7] "결과 복사" 버튼 클릭 (F7) → 클립보드 복사 완료 토스트
     ↓
[8] (선택) 글 수정 후 다시 [4]로 — 재제출 가능
```

## 3. 기능별 명세

### F1. 과제 설명 입력

| 항목 | 내용 |
|---|---|
| 목적 | AI가 "이 과제가 무엇을 요구하는지" 알게 한다. rubric §과제이해 채점의 기준값. |
| 입력 필드 | ① 학교급·학년 (드롭다운, 필수) ② 과목 (드롭다운, 필수) ③ 글 장르 (드롭다운, 필수) ④ 목표 분량 (숫자, 선택) ⑤ 과제문/조건 (텍스트area, 필수) |
| 학교급·학년 옵션 | 중1 / 중2 / 중3 / 고1 / 고2 / 고3 |
| 과목 옵션 | 국어 / 사회 / 역사 / 도덕·윤리 / 과학 / 기타 |
| 글 장르 옵션 | 설명문 / 논설문·주장하는 글 / 감상문·독후감 / 성찰문·수필 / 보고서 / 요약문 / 기타 |
| 프리셋 (F09) | 중2 국어 / 중3 사회 / 고1 국어 등 자주 쓰는 학교급·과목 조합을 원클릭 선택해 ①②③ 자동 채움. 프리셋 데이터는 2022 개정 교육과정 표에서 자동 추출 + 5개 대표 케이스 수동 검수 (기능제안서 §7.2 Q2 결정) |
| 목표 분량 단위 | 글자 수 (예: 800) — 비우면 "제한 없음" |
| 목표 분량 입력 제한 | 50자 이상 2,000자 이하 (본문 입력 제한과 동일 범위). 비우면 제한 없음, 범위 밖 입력 시 E10 처리 |
| 과제문 입력 제한 | 10자 이상 1,000자 이하 |
| 검증 규칙 | 필수 4개(①②③⑤) 미입력 시 F3 버튼 비활성 |
| 완료 기준 | 5개 필드가 출력 JSON의 `assignment` 객체로 정확히 직렬화된다 |

### F2. 학생 글 입력

| 항목 | 내용 |
|---|---|
| 목적 | 채점·첨삭 대상 본문 확보. |
| 입력 필드 | 본문 텍스트area 1개 |
| 글자 수 표시 | 입력과 동시에 실시간 카운트. 목표 분량 입력 시 "현재 620자 / 목표 800자" 형태 |
| 입력 제한 | 50자 이상 2,000자 이하 |
| 검증 규칙 | 50자 미만이면 F3 버튼 비활성 + "본문을 50자 이상 입력해 주세요" 안내 |
| 엣지 케이스 | 2,000자 초과 시 입력 차단 + "2,000자까지 첨삭할 수 있어요" 안내 |
| 입력 정규화 (v0.3 신설) | §4.1.1 입력 정규화 룰을 따라 `(하략)`·`<중략>` 등 본문 외 표기를 제거하고 줄바꿈을 정리한다. 학생 mechanics 오류(맞춤법·띄어쓰기·오탈자)는 보존한다. |
| 완료 기준 | 본문이 출력 JSON의 `submission.body` 로 직렬화되고 `submission.char_count` 가 일치한다 |

### F3. AI 첨삭 실행

| 항목 | 내용 |
|---|---|
| 목적 | F1+F2 입력을 AI에 전달해 채점·피드백·수정가이드 1세트를 생성한다. |
| 트리거 | "AI 첨삭 받기" 버튼 클릭 |
| 사전 조건 | F1 필수 4개 + F2 본문 50자 이상 충족 |

| 출력 | §4.2 출력 JSON 스키마 |

| 응답 시간 목표 | 15초 이내 (초과 시에도 로딩 유지, 60초 초과 시 타임아웃 처리) |
| 프롬프트 | AI 첨삭 프롬프트 v0.1 (`08_ai_prompt_v.1_2026-05-19.md`, Week 1 산출물 — v0.3에서 신설) |
| 실패 처리 | §6 에러 케이스 참조 |
| 완료 기준 | 정상 입력 1건에 대해 §4.2 스키마를 100% 만족하는 출력 1건이 반환된다 |

### F4. 5영역 점수 표시

| 항목 | 내용 |
|---|---|
| 목적 | "지금 몇 점짜리 글인지" 한눈에 보여준다. |
| 표시 요소 | ① 총점 (0~100, 큰 숫자) ② 5영역 각 점수 (0~20) ③ 영역별 점수 막대 |
| **5영역 (rubric v0.4)** | **과제 이해 / 내용 충실도 / 구조·논리 / 표현·문장 / 성장 가능성** — 각 20점 |
| 장르 가중치 | v0.1은 글 장르와 무관하게 5영역 동일 가중치(각 20점). 장르별 가중치 분기는 v1 검토 (기능제안서 §8 Out of Scope) |
| 점수 출처 | 출력 JSON `scores` 배열 |
| 표시 규칙 | 총점 = 5영역 합. 합이 100을 넘거나 각 영역이 0~20 범위를 벗어나면 "점수 오류" 표시 후 재시도 유도 |
| 색상 가이드 | 0~9 주의 / 10~14 보통 / 15~20 양호 (색상은 wireframe 단계 결정) |
| **영역 편차 강조 (v0.3 신설)** | 5영역 중 최대-최소 차가 **6점 이상**이면 화면에 "영역 편차가 큽니다" 칩(또는 안내) 노출 + 총점 옆에 작은 안내문 "총점보다 영역별 피드백을 먼저 보세요". rubric v0.4 §3.5 보조 룰·§4 안내문과 정합. 학생이 "총점 61점이라 평범한 글"이라 오해하지 않게, 영역별 막대를 시각적으로 더 강조한다 (색·두께·움직임은 Week 2 design). |
| 완료 기준 | `scores` 5개 항목과 `total_score` 가 화면 수치와 1:1 일치하고, 영역 편차 6점 이상일 때 강조 안내가 노출된다 |

### F5. 영역별 피드백 표시

| 항목 | 내용 |
|---|---|
| 목적 | "왜 그 점수인지" 근거를 영역별로 보여준다. |
| 표시 요소 | 5영역 각각에 대해 ① 잘한 점 1~2문장 ② 고칠 점 1~2문장 |
| 피드백 출처 | 출력 JSON `scores[].feedback_good`, `scores[].feedback_fix` |
| 표시 규칙 | 5영역 모두 잘한 점·고칠 점이 비어 있지 않아야 함. 빈 값이면 "피드백 생성 실패" 표시 |
| 문체 기준 | 학생 대상 존댓말, 1문장 60자 이내 권장, 지적이 아니라 코칭 톤 |
| **fix 정정 항목 수 (v0.3 신설)** | `feedback_fix` 한 문장에 'A → B' 형식 정정 항목은 **최대 3개**까지. 4개 이상은 `revision_guides`로 분리해 표시(rubric v0.4 §2 원칙 12). 화면 카드가 빽빽해지지 않게. |
| v0.1 범위 외 | 교육과정 과목·영역 태깅(F10)은 Month 2 추가 — v0.1 영역별 피드백은 5영역 rubric 기반만 (기능제안서 §7.2 Q3 결정) |
| 완료 기준 | 5영역 × (잘한 점 + 고칠 점) = 10개 피드백 블록이 빠짐없이 렌더된다 |

### F6. 수정 가이드 표시

| 항목 | 내용 |
|---|---|
| 목적 | "그래서 뭐부터 고칠지" 우선순위 액션을 준다. rubric §성장가능성과 직결. |
| 표시 요소 | 우선순위 순서대로 1~3개 액션 카드 (번호 + 한 줄 액션 + 근거 1문장) |
| 출처 | 출력 JSON `revision_guides` 배열 |
| 표시 규칙 | 최소 1개, 최대 3개. 0개면 "수정 가이드 생성 실패" 표시 |
| 문체 기준 | 동사로 끝나는 실행형 문장 (예: "2문단에 구체적 예시를 1개 추가하세요") |
| 완료 기준 | `revision_guides` 1~3개가 우선순위 순서대로 렌더된다 |

| 항목 | 내용 |
|---|---|
| 목적 | 학생이 결과를 노트·과제 파일·메신저에 붙여넣게 한다. |

## 4. 데이터 구조

### 4.1 입력 JSON 스키마 (F3 요청)

```json
{
  "assignment": {
    "school_level": "중2",
    "subject": "국어",
    "genre": "논설문·주장하는 글",
    "target_char_count": 800,
    "prompt_text": "교복 자율화에 대한 자신의 주장을 근거 2개 이상으로 쓰시오."
  },
  "submission": {
    "body": "(학생이 입력한 본문 전체 — §4.1.1 정규화 적용)",
    "char_count": 620
  },
  "meta": {
    "client_version": "v0.1",
    "submitted_at": "2026-05-14T10:30:00+09:00",
    "attempt_no": 1
  }
}
```

| 필드 | 타입 | 필수 | 비고 |
|---|---|---|---|
| `assignment.school_level` | string | Y | §3 F1 옵션 |
| `assignment.subject` | string | Y | §3 F1 옵션 |
| `assignment.genre` | string | Y | §3 F1 옵션 |
| `assignment.target_char_count` | number\|null | N | 50~2,000, 비우면 null |
| `assignment.prompt_text` | string | Y | 10~1,000자 |
| `submission.body` | string | Y | 50~2,000자, §4.1.1 정규화 후 기준 |
| `submission.char_count` | number | Y | 정규화 후 body 글자 수와 일치 |
| `meta.attempt_no` | number | Y | 재제출 시 +1 |

#### 4.1.1 입력 정규화 룰 (v0.3 신설)

학생이 붙여넣는 본문은 출처(학습지 발췌·문서 스캔·메신저 복사 등)에 따라 채점 대상이 아닌 표기가 섞일 수 있다. F3 호출 전 클라이언트가 다음 룰로 정규화한다.

| 유형 | 처리 | 근거 |
|---|---|---|
| **부분 발췌 표기** `(하략)` · `(중략)` · `(전략)` · `(이하 생략)` (괄호 한자/한글 표기 포함) | 표기와 그 앞뒤 한 칸 공백을 **제거** | 채점 대상은 학생이 실제로 쓴 분량만 (rubric v0.4 §2 원칙 6) |
| **부분 발췌 표기** `<중략>` · `<생략>` · `[중략]` · `…(중략)…` | 표기 제거 후 앞 문장 종결부에 마침표가 없으면 보강하고 뒤 문장 첫 글자 앞에 공백 1칸 | 자연 연결 |
| **줄 끝 줄바꿈 (전송 wrap)** — 단어 중간의 `관\n한`, `감아\n야` 같은 잘림 | **공백 한 칸으로 정규화** (단, 빈 줄(`\n\n`)은 문단 구분으로 **보존**) | 학생 의도가 아닌 전송 잡음. 빈 줄은 학생의 문단 의도일 가능성 보존 |
| **앞뒤 공백·탭** | `trim` | 안전 정규화 |
| **연속 공백 3개 이상** | 공백 1개로 축소 | 안전 정규화 |
| **학생 mechanics 오류** (맞춤법·띄어쓰기·오탈자·비문) | **보존** | 채점 대상. AI가 §3.4 표현·문장에서 판정 |

**원칙**: 정규화는 '학생이 쓴 글이 아닌 표기'만 제거한다. 학생의 글 자체는 어떤 형태로든 변경하지 않는다. 정규화 결과 본문이 50자 미만이면 E2 처리.

**구현 메모**: 정규화는 F2 단계(클라이언트)에서 본문을 채점용 JSON으로 직렬화할 때 적용. 화면 텍스트area에 학생이 본 글은 정규화 전 원본을 유지(학생이 자기 글을 못 알아보면 안 됨).

| E1 | F1 필수 필드 미입력 | F3 버튼 비활성, 미입력 필드 빨간 테두리 |

| E8 | 업스트림(Anthropic) 장애 → 503 | **"일시적 오류예요. 잠시 후 다시 시도해 주세요"** + 재시도 버튼 (사용자 인터넷 탓 아님. 진짜 클라이언트 오프라인은 FE fetch reject 경로에서 "인터넷 연결을 확인…"로 별도 안내 — EPO 2026-05-26 ①, 12 §5.2) |

> **E2/E11(422) vs E3(400) HTTP 코드 구분 (의도됨, EPO 2026-05-26 ②)**: 본문 길이 위반이지만 코드가 다른 것은 의도다. **E3(2,000자 초과)는 정규화 *전* 입력 범위 위반 → 400** / **E2·E11(50자 미만)은 정규화 *후* 채점 불가 확정(의미 판정) → 422**. 규칙: *범위 위반=400 / 정규화 후 처리 불가=422*. (서버 계약 12 §5.2와 정합.)

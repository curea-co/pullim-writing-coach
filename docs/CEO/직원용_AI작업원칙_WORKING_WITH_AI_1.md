# AI와 일하는 법 — 작게 시키지 말고, 야심을 위임하라

> 풀림 직원용 작업 원칙 · 2026-06-01 · 공유본
> **한 줄 요약**: 우리의 산출물이 작은 건 AI가 작아서가 아니라, **우리의 요청이 작아서**다. AI에게 *일*을 시키지 말고 *야심*을 위임하라.

---

## 0. 이 문서를 쓰는 이유 (답답함의 정체)

우리는 AI를 많이 씁니다. 그런데 나오는 산출물의 **스코프가 너무 작습니다.**

- 함수 하나, 문단 하나, 화면 하나 — AI를 **"빠른 타자기"**로 쓰고 있습니다.
- "이 버튼 색 바꿔줘", "이거 요약해줘", "이 함수 만들어줘" 수준에서 멈춥니다.
- 그래서 결과물이 *부분의 합*에 갇히고, 정작 **소비자가 진짜 원하는 경험**까지 못 갑니다.

문제는 AI의 능력이 아니라 **우리가 던지는 질문의 크기**입니다.

> ⭐ **작은 산출물은 작은 프롬프트의 거울이다.**
> AI가 작게 답했다면, 십중팔구 당신이 작게 물었다.

---

## 1. 시대가 바뀌었다 — 병목이 '손'에서 '머리'로 옮겨갔다

| | 예전 (사람이 만들던 시대) | 지금 (AI와 만드는 시대) |
|---|---|---|
| 병목 | **사람의 시간·손** | **사람의 안목·판단** |
| 합리적 전략 | MVP, 최소 구현, 점진주의 | **전체를 보고 전체를 설계** |
| 만드는 순서 | 부분 → 쌓아서 합 | **전체상 → 분해 → 우선순위** |
| 비싼 것 | 만드는 것(실행) | 무엇을 만들지 아는 것(방향) |
| 직원의 핵심 역량 | 빨리 만들기 | **무엇이 진짜 필요한지 알기 + 야심을 키우기** |

예전엔 "일단 작게, 빨리"가 옳았습니다. 사람 시간이 비쌌으니까요.
지금은 **실행 비용이 무너졌습니다.** 5가지 대안을 만드는 비용, 전체 시스템을 한 번에 그리는 비용, 출시 수준까지 밀어붙이는 비용이 예전의 1/10도 안 됩니다.

> 그래서 이제 진짜 질문은 *"얼마나 만들 수 있나"*가 아니라 → **"소비자에게 정말 필요한 경험이 무엇인지 우리가 아는가"**입니다.
> MVP가 목표인 시대는 지났습니다. **전체를 보고, 전체를 만들 수 있는 시대**가 왔습니다.

---

## 2. 열 가지 원칙

**① 요청의 단위를 키워라.**
"함수 하나" 대신 "이 기능 전체 + 엣지케이스 + 에러처리 + 테스트". "한 화면" 대신 "전체 흐름". 일을 잘게 쪼개 주지 말고, **목표(원하는 경험)를 주고 쪼개는 것까지 맡겨라.**

**② MVP가 아니라 '전체상'부터 그려라.**
전체를 먼저 보고, *그다음에* 무엇을 먼저 낼지 정하라. 부분부터 쌓으면 부분의 합에 영원히 갇힌다. **전체 → 분해**지, 부분 → 합이 아니다.

**③ 하나의 답이 아니라 여러 대안을 받아라.**
AI는 5가지 접근을 거의 같은 비용으로 낸다. 첫 답에 만족하지 마라. *"다른 3가지 방식으로, 각각의 트레이드오프와 추천까지."* **발산 후 수렴.**

**④ 소비자 경험에서 거꾸로 설계하라.**
"무엇을 만들까"가 아니라 **"사용자가 어떤 순간에 무엇을 느껴야 하나"**에서 시작하라. 기능 목록이 아니라 경험을 먼저.

**⑤ 패러다임을 의심하라.**
"원래 이렇게 한다"는 제약은 대부분 *옛 비용 구조의 잔재*다. "이 폼이 꼭 필요한가? 이 단계가 꼭 필요한가? 더 좋은 방식은 없나?"를 매번 물어라.

**⑥ 산출물은 끝이 아니라 시작이다.**
결과가 나오면 멈추지 말고 되물어라. *"이걸 2배 좋게 하려면? 절반 시간에 하려면? 사용자가 더 적게 일하게 하려면?"* 더 빠르게, 더 좋게, 더 효과적으로 — 끝없이.

**⑦ AI를 실행자가 아니라 사고 파트너로 써라.**
받아쓰기 말고 논쟁시켜라. *"이 계획의 가장 큰 약점은? 내가 놓친 건? 더 야심차게 한다면 어떻게?"* AI가 당신을 밀어붙이게 하라.

**⑧ 완성도까지 밀어붙여라.**
"대충 초안"에서 멈추지 말고 **"출시 가능한 수준 + 검증"**까지. 스코프만이 아니라 *깊이*도 키워라.

**⑨ 야심에는 검증이 따라야 한다.**
크게 만들수록 틀릴 위험도 크다. AI에게 *스스로 반증·테스트·검증*을 시켜라. **야심과 검증은 한 쌍이다.** (큰 거짓말은 큰 손해다.)

**⑩ 취향과 판단은 끝까지 네 몫이다.**
AI가 양과 속도를 책임지면, 사람은 **방향·취향·"이게 진짜 필요한가"**를 책임진다. 그게 이제 직원의 진짜 일이다.

## 3. 이렇게 일하지 마라 / 이렇게 일하라 (Before → After)

> 똑같은 일을 *작게* 시키는 법과 *크게* 시키는 법. 차이는 능력이 아니라 **요청의 야심**이다.

| 분야 | ❌ 작게 (타자기) | ✅ 크게 (야심 위임) |
|---|---|---|
| 개발 | "로그인 함수 만들어줘" | "전체 인증 경험을 설계하고, 엣지케이스·에러·보안·테스트까지 구현해. 기존 코드와 통합 방식도 설명하고, 1주차에 낼 범위를 추천해줘" |
| 기획 | "이 화면에 입력 박스 추가해줘" | "이 입력 경험을 처음부터 다시 설계해. 지금 패러다임을 깨서 사용자가 더 적게 일하게 만들 3가지 방향과 추천을 줘. 내가 놓친 사용자 시나리오도 찾아줘" |
| 디자인 | "버튼 색 바꿔줘" | "이 화면에서 전환율을 떨어뜨리는 요소를 전부 찾아 고치고, 더 나은 레이아웃 대안 3개를 트레이드오프와 함께 보여줘" |
| 문서 | "이 보고서 요약해줘" | "이 보고서를 임원용·실무용·고객용 3개 버전으로, 각 독자의 의사결정에 맞게 다시 써줘. 빠진 논점도 지적해줘" |
| 분석 | "이 숫자 평균 내줘" | "이 데이터에서 우리가 놓치고 있는 패턴·이상치·기회를 찾고, 다음에 무엇을 해야 하는지 근거와 함께 제안해줘" |
| 마케팅 | "인스타 문구 써줘" | "이 제품의 핵심 가치를 정의하고, 채널별(인스타·블로그·세일즈) 메시지 체계를 만든 뒤, A/B로 시험할 카피 변형까지 줘" |

**패턴이 보이나요?** ❌는 *결과물*을 지정하고, ✅는 *목표와 야심의 수준*을 준 뒤 나머지를 위임합니다.

## 4. 실제 작업 루프 (방법)

```
[1] 크게 연다     → 맥락 + 목표 + 제약 + 성공기준을 한 번에 (얇은 프롬프트 금지)
[2] 발산한다      → "다른 N가지 방식으로, 트레이드오프와 추천까지"
[3] 전체를 본다   → "전체상을 먼저 그리고, 그중 1차 출시 범위를 추천해줘"
[4] 공격한다      → "이 안의 가장 큰 약점은? 내가 놓친 건?" (레드팀)
[5] 되묻는다      → "이걸 더 빠르게 / 더 좋게 / 더 효과적으로 하려면?"
[6] 검증을 박는다 → 실제 실행·테스트·사용자 관점 점검을 AI에게 시킨다
[7] 네가 정한다   → 방향·취향·"진짜 필요한가"는 사람이 판단
```

- **큰 작업일수록 맥락을 아끼지 마라.** 배경·목표·제약·성공 기준을 충분히 주면 산출이 커진다. 맥락 한 줄이 결과의 크기를 정한다.
- **여러 버전을 병렬로** 만들어 비교하라. 하나를 다듬는 것보다 셋을 만들어 고르는 게 낫다.
- **완료 기준을 높게 잡아라.** "되는 수준"이 아니라 "출시 수준 + 검증 통과"를 기본값으로.

## 5. 매일 쓰는 체크리스트

**작업을 시작하기 전 (스스로에게)**
- [ ] 나는 *결과물*을 지정했나, 아니면 *목표와 야심*을 줬나?
- [ ] 전체상을 먼저 그렸나, 부분부터 시켰나?
- [ ] 소비자가 어떤 경험을 느껴야 하는지에서 출발했나?
- [ ] "원래 이렇게 한다"는 제약을 한 번이라도 의심했나?

**산출물을 받은 후 (멈추기 전에)**
- [ ] 이게 *최선*인가, 아니면 *첫 번째 답*인가?
- [ ] "2배 좋게 / 절반 시간에" 하려면 어떻게 물어야 하나?
- [ ] 내가 놓친 사용자·엣지케이스·논점은 없나? (AI에게 물어봤나?)
- [ ] 야심만큼 검증했나? (테스트·반증·실제 실행)

## 6. 프롬프트에 박아 넣을 문장 (복붙용)

- "**최소가 아니라 최대로.** 전체를 보고 설계해줘."
- "**다른 3가지 방식**으로, 각각의 트레이드오프와 네 추천까지."
- "전체상을 먼저 그리고, 그중 **먼저 낼 범위를 추천**해줘."
- "이 계획의 **가장 큰 약점을 공격**해봐. 내가 놓친 건?"
- "이걸 **2배 좋게 / 절반 시간에** 하려면?"
- "**출시 가능한 수준**까지, 그리고 **스스로 검증**해줘."
- "사용자가 **더 적게 일하게** 만들 방법은?"
- "**내가 놓치는 게 또 있지 않을까?**"

> 마지막 문장은 특히 강력합니다 — 발산이 한 번 더 일어나고, 사각지대가 드러납니다.

---

## 7. 균형 — 야심에는 두 가지 안전장치가 따른다

1. **검증.** 크게 만들수록 틀리면 크게 틀린다. AI에게 *반증·테스트·실제 실행*을 시켜 야심을 떠받쳐라. 야심 없는 검증은 소심하고, 검증 없는 야심은 위험하다.
2. **판단.** AI는 무엇이든 *그럴듯하게* 많이 만든다. **"이게 진짜 소비자에게 필요한가?"**를 묻는 건 끝까지 사람의 일이다. 양과 속도는 AI가, 방향과 취향은 네가.

---

## 8. 북극성 (한 줄)

> **AI에게 일을 시키지 말고, 야심을 위임하라.**
> 그리고 그 야심이 *옳은 방향인지*는, 끝까지 네가 판단하라.

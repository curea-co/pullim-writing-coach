# AI와 일하는 법 — 작게 시키지 말고, 야심을 위임하라

> 풀림 직원용 작업 원칙 · 2026-06-01 · v2.0 (근거 보강판) · 공유본
> **한 줄 요약**: 우리의 산출물이 작은 건 AI가 작아서가 아니라 **우리의 요청이 작아서**다. AI에게 *일*을 시키지 말고 *야심*을 위임하라 — 단, 방향은 네가 판단하고 검증은 안에 박아 넣어라.

---

## 0. 이 문서를 쓰는 이유 (답답함의 정체)

우리는 AI를 많이 씁니다. 그런데 나오는 산출물의 **스코프가 너무 작습니다.**

- 함수 하나, 문단 하나, 화면 하나 — AI를 **"빠른 타자기"**로 쓰고 있습니다.
- "이 버튼 색 바꿔줘", "이거 요약해줘", "이 함수 만들어줘" 수준에서 멈춥니다.
- 그래서 결과물이 *부분의 합*에 갇히고, 정작 **소비자가 진짜 원하는 경험**까지 못 갑니다.

원인은 AI의 능력이 아니라 **우리가 던지는 질문의 크기**입니다.

> ⭐ **작은 산출물은 작은 프롬프트의 거울이다.**
> AI가 작게 답했다면, 십중팔구 당신이 작게 물었다.
> *(단, 모든 작은 산출물이 프롬프트 탓은 아니다. 때로는 작은 게 정답이고, 때로는 AI가 '마지막 30%' 벽에 막힌 것이다 — §3, §5 참고.)*

## 1. 무엇이 바뀌었고, 무엇이 안 바뀌었나 (논리의 출발점)

가장 흔한 오해부터 정리합시다. **"이제 MVP는 끝났다"는 말은 절반만 맞습니다.**

- **무너진 것 — 만드는 비용(실행).** 5가지 대안을 만들고, 전체 시스템을 한 번에 그리고, 출시 수준까지 밀어붙이는 비용이 예전의 1/10도 안 됩니다. 코드를 *영어로* 쓰는 시대(카파시의 "Software 3.0")가 실제로 왔습니다.
- **안 무너진 것 — 틀리는 비용(방향).** *무엇이 소비자에게 진짜 필요한지*를 아는 일은 여전히 비쌉니다. MVP의 본래 목적은 "적게 만들기"가 아니라 **"싸게 배우기"**였습니다. 그 목적은 지금도 유효합니다.

> 그래서 정확한 명제는 이것입니다:
> **"전체를 만들 수 있다"가 "아무거나 크게 만들어라"는 아니다.**
> 만드는 비용이 무너졌으니 **전체를 만들어 놓고도 빨리 배울 수 있다.** 단, *틀린 것을 크게 만드는 손해*는 예전보다 더 큽니다. → 그래서 **야심은 항상 '판단'과 '검증'과 한 묶음**으로 움직여야 합니다.

병목이 **'손'에서 '머리'로** 옮겨갔다는 뜻입니다. 이제 비싼 것은 실행이 아니라 *방향·취향·검증*입니다.

| | 예전 (사람이 만들던 시대) | 지금 (AI와 만드는 시대) |
|---|---|---|
| 병목 | 사람의 시간·손 | **사람의 안목·판단·검증** |
| 만드는 순서 | 부분 → 쌓아서 합 | **전체상 → 분해 → 우선순위** |
| MVP의 의미 | 적게 만들어 배우기 | **전체를 만들어 놓고도 빨리 배우기** |
| 비싼 것 | 만드는 것(실행) | 무엇을 만들지 아는 것 + 틀리지 않았는지 검증하는 것 |
| 직원의 핵심 역량 | 빨리 만들기 | **야심을 키우고 · 방향을 판단하고 · 검증을 설계하기** |

## 2. 현장 증거 — 2025~2026 실제로 이렇게 일하고 있다

추상론이 아닙니다. 지금 업계 표준이 어디까지 와 있는지부터 봅시다.

**▷ 야심의 기준선이 올라갔다.**
- **Shopify** (CEO 토비 뤼트케, 2025-04 사내 메모): *"반사적 AI 사용은 이제 기본 기대치다(Reflexive AI usage is now a baseline expectation)."* 인력·예산을 더 요청하기 전에 **"왜 AI 에이전트가 이 일을 못 하는지"를 먼저 증명**해야 한다(이른바 "뤼트케 테스트").
- **Duolingo** (CEO 루이스 폰 안, 2025-04): 회사를 **"AI-first"**로 선언. AI가 할 수 있는 일은 외주를 줄이고, 팀이 증원을 원하면 *"왜 AI로는 안 되는지"*를 먼저 입증.
→ 시사점: **"AI로 어디까지 가능한가"를 먼저 끝까지 밀어보는 것이 기본값**이 되었다. 작게 쓰는 건 이제 *덜 쓰는 것*이다.

**▷ 패러다임 자체가 이동했다.**
- 안드레이 카파시: **"Software 3.0 — 영어가 프로그래밍 언어"**(2025 키노트). 2025년 초 "바이브 코딩"을 만들었고(콜린스 사전 2025 올해의 단어), 최근엔 *"바이브 코딩은 한물갔다. 이제 LLM 에이전트로 짜는 게 전문가의 기본 워크플로 — 단 **감독과 검증을 더 붙여서**"*라고 정정했습니다.
→ 시사점: **크게 위임하되, 감독·검증을 더 촘촘히.** 이게 프로의 방식.

**▷ "전체 → 분해"가 도구로 제품화됐다 (스펙 주도 개발).**
- **GitHub Spec Kit**, **Amazon Kiro**(2025-07, *"from vibe coding to viable code"*): 코드를 바로 짜지 않고 **명세(스펙)를 먼저** 만든 뒤 `명세 → 설계 → 작업 분해 → 구현` 순으로 간다. **스펙이 진실의 원천(source of truth).**
→ 시사점: 우리가 §3에서 말할 "**전체상부터 그리고 분해하라**"가 이미 업계 표준 워크플로다.

**▷ 검증이 곧 경쟁력이다.**
- 하멜 후사인 *"Your AI Product Needs Evals"*: **실패하는 LLM 제품의 공통 근본 원인은 '제대로 된 평가(eval) 체계의 부재'.** "evals를 100% 통과한다면 시스템을 충분히 몰아붙이지 않은 것"이라고까지 말합니다.
- Anthropic **Claude Code 베스트 프랙티스**: *"탐색 → 계획 → 구현 → 커밋"*, **TDD가 가장 강력한 패턴**, 그리고 *"성공했다고 주장하지 말고 증거(테스트 출력·실행 결과)를 보여라."*
→ 시사점: **야심의 안전벨트는 검증이다.** 검증은 브레이크가 아니라, *액셀을 끝까지 밟게 해주는 장치*다.

**▷ 그러나 — 과신은 측정으로 반박됐다 (정직한 한계).**
- **METR 무작위대조연구**(2025-07): 숙련 오픈소스 개발자 16명이 *익숙한 성숙 코드베이스*에서 AI를 쓰자 오히려 **19% 느려졌다.** 본인들은 24% 빨라질 거라 예측했고, 쓰는 동안엔 20% 빨라졌다고 *느꼈지만*, 실제는 반대였다. (체감 ≠ 실제.)
- 애디 오스마니 **"70% 문제"**: AI는 70~80%까지 순식간에 데려다주지만, **마지막 30%(엣지케이스·보안·운영 통합)는 여전히 사람의 몫**이고 *프로토타입과 출시 제품의 간극은 좁혀지지 않았다.*
→ 시사점: **크게 시키되, 마지막 30%와 '내가 빨라진 느낌'을 의심하라.** 야심과 검증을 같이 키워야 하는 이유가 여기 있다.

## 3. 열 가지 원칙

**① 요청의 단위를 키워라.**
"함수 하나" 대신 "이 기능 전체 + 엣지케이스 + 에러처리 + 테스트". "한 화면" 대신 "전체 흐름". 일을 잘게 쪼개 주지 말고, **목표(원하는 경험)를 주고 쪼개는 것까지 맡겨라.**

**② MVP가 아니라 '전체상'부터 그려라.**
전체를 먼저 보고 *그다음에* 무엇을 먼저 낼지 정하라. 부분부터 쌓으면 부분의 합에 갇힌다. **전체 → 분해**지, 부분 → 합이 아니다. (= 스펙 주도 개발의 핵심.)

**③ 하나의 답이 아니라 여러 대안을 받아라.**
AI는 5가지 접근을 거의 같은 비용으로 낸다. 첫 답에 만족하지 마라. *"다른 3가지 방식으로, 각각의 트레이드오프와 추천까지."* **발산 후 수렴.**

**④ 소비자 경험에서 거꾸로 설계하라.**
"무엇을 만들까"가 아니라 **"사용자가 어떤 순간에 무엇을 느껴야 하나"**에서 시작하라. 기능 목록이 아니라 경험을 먼저.

**⑤ 패러다임을 의심하라.**
"원래 이렇게 한다"는 제약은 대부분 *옛 비용 구조의 잔재*다. "이 폼이 꼭 필요한가? 이 단계가 꼭 필요한가? 더 좋은 방식은 없나?"를 매번 물어라.

**⑥ 산출물은 끝이 아니라 시작이다.**
결과가 나오면 멈추지 말고 되물어라. *"이걸 2배 좋게 하려면? 절반 시간에 하려면? 사용자가 더 적게 일하게 하려면?"* 더 빠르게, 더 좋게, 더 효과적으로 — 끝없이.

**⑦ AI를 실행자가 아니라 사고 파트너로 써라.**
받아쓰기 말고 논쟁시켜라. *"이 계획의 가장 큰 약점은? 내가 놓친 건? 더 야심차게 한다면?"* 카파시의 표현대로, 프로의 방식은 *"위임 + 더 많은 감독과 검증"*이다.

**⑧ 맥락은 '많이'가 아니라 '정확히' 줘라.**
Anthropic의 컨텍스트 엔지니어링 원칙: *"핵심을 정확히 담은 400토큰 파일이, 이것저것 다 담은 4,000토큰 파일을 이긴다."* 배경·목표·제약·성공 기준을 **충분하되 군더더기 없이** 줘라. 맥락의 질이 산출의 크기를 정한다.

**⑨ 검증을 안에 박아 넣어라 (야심의 안전벨트).**
크게 만들수록 틀리면 크게 틀린다. AI에게 *스스로 반증·테스트·eval*을 시켜라. *"성공했다 말하지 말고 증거를 보여줘."* **검증은 브레이크가 아니라 액셀을 끝까지 밟게 해주는 장치다.**

**⑩ 취향과 판단은 끝까지 네 몫이다.**
AI가 양과 속도를 책임지면, 사람은 **방향·취향·"이게 진짜 필요한가"**를 책임진다. METR이 보여줬듯 *체감 생산성은 거짓말을 한다* — 그래서 마지막 판단은 사람이 한다.

## 4. 언제 '최대로', 언제 '신중하게' — 야심의 조절 규칙

무조건 크게가 아니다. **야심은 '가역성'과 '위험'에 맞춰 조절**한다.

| 상황 | 모드 | 이유 |
|---|---|---|
| 가역적·탐색적 (목업, 초안, 아이디어, 새 기능 실험) | **🟢 최대로** | 틀려도 버리면 그만. 발산·전체 구현·여러 대안의 이득이 압도적 |
| 부분적 위험 (출시 예정 기능, 고객 대면 카피) | **🟡 크게 + 검증** | 크게 만들되 eval·테스트·리뷰를 통과시킨 뒤 내보냄 |
| 비가역적·고위험 (성숙 코드 대수술, 결제·개인정보, 대외 발표) | **🔴 구조 먼저, 신중하게** | METR 경고 구간. 스펙·계획·검증을 먼저, 그다음 실행 |

> 규칙: **버릴 수 있는 곳에선 최대로 질러라. 되돌릴 수 없는 곳에선 스펙과 검증을 먼저 깔아라.**

## 5. 이렇게 일하지 마라 / 이렇게 일하라 (Before → After)

> 똑같은 일을 *작게* 시키는 법과 *크게* 시키는 법. 차이는 능력이 아니라 **요청의 야심**이다.

| 분야 | ❌ 작게 (타자기) | ✅ 크게 (야심 위임 + 검증) |
|---|---|---|
| 개발 | "로그인 함수 만들어줘" | "전체 인증 경험을 설계하고, 엣지케이스·에러·보안·테스트까지 구현해. 기존 코드 통합 방식도 설명하고, **테스트 출력으로 증명**한 뒤 1주차 출시 범위를 추천해줘" |
| 기획 | "이 화면에 입력 박스 추가해줘" | "이 입력 경험을 처음부터 다시 설계해. 지금 패러다임을 깨서 사용자가 더 적게 일하게 만들 3가지 방향과 추천을 줘. **내가 놓친 사용자 시나리오도** 찾아줘" |
| 디자인 | "버튼 색 바꿔줘" | "이 화면에서 전환율을 떨어뜨리는 요소를 전부 찾아 고치고, 더 나은 레이아웃 대안 3개를 트레이드오프와 함께 보여줘" |
| 문서 | "이 보고서 요약해줘" | "이 보고서를 임원용·실무용·고객용 3개 버전으로, 각 독자의 의사결정에 맞게 다시 써줘. **빠진 논점도** 지적해줘" |
| 분석 | "이 숫자 평균 내줘" | "이 데이터에서 우리가 놓치는 패턴·이상치·기회를 찾고, 다음에 무엇을 해야 하는지 근거와 함께 제안해줘. **반대 해석도** 한 번 해줘" |
| 마케팅 | "인스타 문구 써줘" | "이 제품의 핵심 가치를 정의하고, 채널별 메시지 체계를 만든 뒤, A/B로 시험할 카피 변형까지 줘" |

**패턴이 보이나요?** ❌는 *결과물*을 지정하고, ✅는 *목표·야심의 수준·검증*을 준 뒤 나머지를 위임합니다.

## 6. 실제 작업 루프 (업계 표준 = 우리 표준)

> Anthropic의 *"탐색 → 계획 → 구현 → 커밋"* + 스펙 주도 개발 + evals를 우리 식으로.

```
[1] 크게 연다     → 맥락 + 목표 + 제약 + 성공기준을 정확히 (얇은 프롬프트 금지, 군더더기도 금지)
[2] 발산한다      → "다른 N가지 방식으로, 트레이드오프와 추천까지"
[3] 전체를 본다   → "전체상(스펙)을 먼저 그리고, 그중 1차 출시 범위를 추천해줘"
[4] 공격한다      → "이 안의 가장 큰 약점은? 내가 놓친 건?" (레드팀)
[5] 되묻는다      → "더 빠르게 / 더 좋게 / 더 효과적으로 하려면?"
[6] 증거를 받는다 → "성공했다 말하지 말고 테스트·실행 결과로 증명해" (eval/검증 내장)
[7] 네가 정한다   → 방향·취향·"진짜 필요한가"는 사람이 판단 (체감 생산성 불신)
```

- **맥락은 질로 승부한다.** 더 많이가 아니라 더 정확히(컨텍스트 엔지니어링).
- **여러 버전을 병렬로** 만들어 고르는 게, 하나를 다듬는 것보다 낫다.
- **완료 기준을 높게.** "되는 수준"이 아니라 "출시 수준 + 검증 통과"가 기본값.
- **체감을 믿지 마라.** "빨라진 느낌"은 측정으로 확인하기 전엔 가설일 뿐(METR).

---

## 7. 매일 쓰는 체크리스트

**작업을 시작하기 전**
- [ ] 나는 *결과물*을 지정했나, *목표와 야심*을 줬나?
- [ ] 전체상을 먼저 그렸나, 부분부터 시켰나?
- [ ] 소비자가 어떤 경험을 느껴야 하는지에서 출발했나?
- [ ] 이건 가역적인가(최대로) / 비가역적인가(검증 먼저)? (§4)

**산출물을 받은 후 (멈추기 전에)**
- [ ] 이게 *최선*인가, *첫 번째 답*인가?
- [ ] "2배 좋게 / 절반 시간에" 하려면 어떻게 물어야 하나?
- [ ] 내가 놓친 사용자·엣지케이스·논점은? (AI에게 물어봤나?)
- [ ] 야심만큼 검증했나? 마지막 30%는 누가 책임지나? (테스트·증거·실제 실행)

---

## 8. 프롬프트에 박아 넣을 문장 (복붙용)

- "**최소가 아니라 최대로.** 전체를 보고 설계해줘."
- "**다른 3가지 방식**으로, 각각의 트레이드오프와 네 추천까지."
- "전체상(스펙)을 먼저 그리고, 그중 **먼저 낼 범위를 추천**해줘."
- "이 계획의 **가장 큰 약점을 공격**해봐. 내가 놓친 건?"
- "이걸 **2배 좋게 / 절반 시간에** 하려면?"
- "**성공했다고 말하지 말고, 테스트·실행 결과로 증명**해줘."
- "사용자가 **더 적게 일하게** 만들 방법은?"
- "**내가 놓치는 게 또 있지 않을까?**"

> 마지막 문장은 특히 강력합니다 — 발산이 한 번 더 일어나고 사각지대가 드러납니다.

---

## 9. 자주 하는 오독 (이렇게 읽으면 틀린다)

- ❌ *"무조건 크게 만들라는 거구나."* → 아니다. **가역적인 곳에선 최대로, 비가역적인 곳에선 검증 먼저**(§4).
- ❌ *"맥락을 최대한 많이 넣으라는 거구나."* → 아니다. **정확히, 핵심만**(§3-⑧).
- ❌ *"AI가 다 해주니 사람은 빠져도 되겠네."* → 정반대. 마지막 30%·방향·검증이 **사람의 일로 더 중요해졌다**(70% 문제·METR).
- ❌ *"내가 빨라진 느낌이면 잘하는 거지."* → 체감은 거짓말을 한다. **증거로 확인하라.**

---

## 10. 북극성 (한 줄)

> **AI에게 일을 시키지 말고, 야심을 위임하라.**
> 그리고 그 야심을 — **방향은 네가 판단하고, 검증은 안에 박아 넣어** — 끝까지 책임져라.
>
> *위임 + 판단 + 검증.* 이 셋이 한 묶음일 때만 "전체를 만드는 시대"가 우리 편이 된다.
